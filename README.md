# CUMT-计算机本科生毕业论文-Skill

面向中国矿业大学计算机相关本科毕业设计（论文）场景的模板分析与材料整理技能包。

> **重要声明**
>
> 本仓库**只用于技术分析、材料梳理、结构核对、版式约束整理与证据清单构建**。  
> **不建议直接用于生成、拼接、润色或替代撰写毕业论文正文**，更**不能**作为代写、洗稿、伪造研究内容或规避学术审查的工具。  
> 论文正文、研究结论、实验结果、图表说明、参考文献使用与最终提交，必须由学生本人在导师指导下独立完成并逐条核验。

## 项目定位

本仓库的定位不是“帮你把论文写完”，而是帮助你把与论文相关的约束条件和真实项目材料整理清楚，避免在毕业设计（论文）过程中出现以下常见问题：

- 学校模板、任务书、样文之间要求不一致
- 论文结构不清楚，不知道哪些章节必须有
- 项目源码、数据库表、页面截图、流程图、测试结果无法对应到论文章节
- 参考文献、图表、附录、外文翻译等材料准备混乱

因此，本仓库强调的是：

1. **学校模板优先**
2. **项目事实优先**
3. **证据链优先**
4. **学术诚信优先**

## 强调使用边界

### 本仓库适合做什么

- 分析中国矿业大学毕业设计（论文）模板的章节顺序和格式要求
- 对照任务书、开题报告、样文，整理论文的必备结构
- 从真实项目源码、数据库结构、接口文档、测试记录中提炼“项目事实底稿”
- 生成论文写作前的目录建议、章节字数预算和图表清单
- 识别学校模板、样文和任务要求之间的冲突项
- 整理截图、E-R 图、流程图、表格、附录等证据材料
- 在已有草稿的前提下检查是否缺少必要章节、必要图表或必要证据

### 本仓库不适合做什么

- 不适合作为“整篇论文自动生成器”
- 不适合直接产出可提交的成品论文并绕过本人写作
- 不适合捏造实验数据、系统截图、测试结果、参考文献或代码实现
- 不适合替代导师意见、院系规范和学校最终审核要求
- 不适合用于降低原创性、规避查重或掩盖学术不端

如果你的目标是“尽快生成一篇能交的论文正文”，那么这不是本仓库推荐的使用方式。  
本仓库更适合在**写作前做分析**、在**写作中做校对**、在**定稿前做合规检查**。

## 学术诚信声明

请特别注意以下原则：

1. **不得虚构研究内容**  
   不得编造不存在的系统模块、数据库表、接口、测试结果、业务流程或部署效果。

2. **不得伪造图表和截图**  
   所有流程图、E-R 图、系统截图、实验图表、测试表格，都应能追溯到真实项目或真实研究材料。

3. **不得伪造参考文献**  
   不得凭空生成作者、题名、期刊、年份、DOI 或链接；不确定真实性的文献宁可不用。

4. **不得代替独立写作**  
   论文正文应由学生本人在理解项目背景、实现细节、相关文献和研究边界的基础上独立完成。

5. **不得绕过导师审核**  
   任何通过本仓库整理出的目录、结构、证据清单或格式建议，都只能作为辅助结果，最终应服从导师和学院要求。

6. **不得将分析结果直接当作学术成果提交**  
   结构化分析、目录建议、证据清单、格式摘要，不等于研究成果本身。

## 适用对象

本仓库更适合以下人群：

- 正在准备中国矿业大学计算机相关专业本科毕业设计（论文）的学生
- 已有真实项目、系统、平台、数据库课程设计或工程实现，但论文材料尚未整理的人
- 已拿到学校模板、任务书、样文，但结构与格式还没理顺的人
- 需要先做材料分析和章节规划，再开始独立写正文的人

本仓库**不适合**以下诉求：

- 直接替代本人完成整篇论文
- 在缺乏真实项目与真实研究支撑的情况下“快速拼稿”
- 伪造工作量、伪造技术实现或伪造测试结果

## 仓库内容

### 1. `SKILL.md`

主技能说明文件，定义了整体工作方式、输入优先级、项目事实冻结方法、模板适配流程和最终检查要点。

### 2. `assets/cumt-thesis-template.doc`

中国矿业大学毕业设计（论文）模板原始资产。  
该文件是学校模板版式的主要依据，使用时应**复制一份工作副本**，不要直接修改原始模板资产。

### 3. `references/template-spec.md`

对学校模板的结构顺序、双语摘要、双语目录、外文翻译、附录位置以及若干高置信格式规则的整理摘要。

### 4. `references/project-first-workflow.md`

面向软件、系统、网站、数据库类毕业设计（论文）的项目型工作流说明，重点包括：

- 如何整理项目事实底稿
- 如何规划章节与字数
- 如何安排系统设计、详细实现、测试验证等章节
- 如何准备截图、流程图、E-R 图、表格和附录

### 5. `agents/openai.yaml`

界面展示相关的名称与描述配置文件，不承载论文事实或学术内容本身。

## 推荐使用流程

本仓库推荐的是“**先分析，再写作；先核验，再定稿**”的流程。

### 第一步：收集输入材料

尽量准备以下材料：

- 学校论文模板
- 任务书 / 开题报告
- 往届样文
- 论文题目
- 字数或页数要求
- 项目源码
- README、数据库设计、接口文档、部署记录
- 测试截图、系统截图、流程说明

如果材料不完整，也应先明确“已有材料”和“缺失材料”，不要一开始就急于成文。

### 第二步：冻结项目事实

在写任何正文前，先整理一份稳定的“项目事实底稿”，至少包括：

- 课题目标
- 研究/实现对象
- 用户角色
- 核心功能模块
- 关键业务流程
- 主要数据实体或数据库表
- 技术栈
- 已实现内容
- 已知限制
- 可用证据材料

后续章节写作应围绕这份底稿展开，避免前后口径不一致。

### 第三步：分析模板与样文

对学校模板、样文、任务书进行结构和格式分析，重点看：

- 论文整体章节顺序
- 哪些页面是必须保留的
- 摘要、关键词、目录、参考文献、致谢的样式
- 一级、二级、三级标题的层级深度
- 图题、表题、代码片段通常出现在哪些章节
- 中英文内容的排版差异

### 第四步：输出结构化分析结果

在正式写正文前，建议先形成以下结果：

- 输入资料清单
- 目录建议
- 各章字数预算
- 样式冲突说明
- 图表与截图需求清单
- 参考文献核验清单

这些内容应作为写作前的准备材料，而不是直接当成论文正文提交。

### 第五步：学生本人独立写作

在前面分析清楚之后，再由学生本人基于：

- 真实项目实现
- 真实研究过程
- 真实参考文献
- 真实测试结果

独立完成正文写作、图表说明与结论表述。

### 第六步：定稿前核对

在论文完成后，再用本仓库对照检查：

- 是否缺章节
- 是否缺图表
- 是否缺截图
- 是否缺数据库设计说明
- 是否缺测试内容
- 是否存在占位文本
- 是否存在格式冲突
- 是否存在可疑引用或无法核验的材料

## 对软件/系统类论文的特别说明

如果论文基于真实的软件系统、数据库平台、管理系统、Web 项目或桌面程序，本仓库尤其适合帮助你整理以下内容：

- 系统总体架构
- 功能模块划分
- 权限与角色设计
- 关键业务流程
- 数据库 E-R 图
- 主要表结构
- 页面截图清单
- 测试用例清单
- 附录中的代码、SQL 或接口说明

但仍要强调：

> **这些内容只能帮助你建立“可写、可证、可核验”的技术分析框架，不能代替你本人完成论文正文。**

## 关于模板与 WPS 的说明

仓库内置的是老式 `.doc` 模板。  
如果你主要使用 WPS，请特别注意：

- 不要频繁手动重设整篇样式
- 优先在已有段落样式上替换文字
- 尽量保留原有分页、分节和目录结构
- 图题、表题、摘要、声明页不要轻易整体重建

模板资产本身应始终视为结构与版式的最高参考。

## 已知局限

1. 老式 `.doc` 模板的部分格式信息不如 `.docx` 易于程序化分析。  
2. WPS 与不同版本 Office 在分页、目录域、题注等细节上可能有兼容差异。  
3. 样文与任务书之间可能存在口径冲突，仍需人工判断。  
4. 参考文献、截图、图表、测试结果的真实性无法仅靠模板规则保证。  
5. 最终论文是否符合学院要求，必须以导师、学院和学校审查为准。

## 不当使用示例

以下做法都不符合本仓库的设计初衷：

- 直接要求生成整篇毕业论文并原样提交
- 用不存在的项目材料拼接所谓“系统实现”
- 用虚构文献填满参考文献列表
- 为了看起来完整而随意补测试结果、流程图或截图
- 用自动生成内容替代本人独立思考、实验和写作

如果你的目标是上述行为，请不要使用本仓库。
