# CUMT-Computer-Undergraduate-Thesis-Skill

面向中国矿业大学计算机相关本科毕业设计（论文）场景的技能包，用于整理真实项目材料、分析学校模板要求、检查论文结构和格式，并帮助建立可核验的技术证据链。

> **重要声明**
>
> 本仓库只用于技术分析、材料梳理、结构核对、格式约束整理、图表规范检查和证据清单构建。<br>
> **不建议直接用于生成、拼接、润色或替代撰写毕业论文正文**，更不能作为代写、洗稿、伪造研究内容、伪造实验数据或规避学术审查的工具。<br>
> 论文正文、实验结果、系统截图、参考文献、研究结论和最终提交版本，必须由学生本人在导师指导下独立完成、理解并逐项核验。

## 项目定位

这个技能包的核心目标不是“把论文写完”，而是帮助学生在写作前和定稿前把关键问题理清楚：

- 学校模板、任务书、样文和导师要求之间是否冲突；
- 论文目录是否符合计算机类本科毕业设计的常见结构；
- 项目源码、数据库表、接口、页面、截图、测试记录能否对应到论文各章节；
- 图、表、公式、参考文献、目录、页码等是否符合毕业论文格式；
- 正文中的功能、数据、实验和结论是否有真实证据支撑；
- 语言是否过度模板化、空泛化，是否需要做保守的自然化和具体化调整。

更简单地说，它适合做“写作前的技术整理”和“提交前的规范核对”，不适合做“自动成文”。

## 学术诚信边界

使用本技能包时必须遵守以下原则：

1. **不编造功能**
   没有在项目源码、页面、接口、数据库或演示环境中出现的功能，不应写成已经实现。

2. **不编造数据**
   演示数据、模拟数据、测试数据和真实教学数据必须区分清楚，不能把原型验证写成大规模真实应用。

3. **不编造图表**
   流程图、E-R 图、系统截图、实验图表都应能追溯到真实系统、真实设计或明确说明的数据来源。

4. **不编造文献**
   不得凭空生成作者、题名、期刊、年份、卷期、页码、DOI 或链接；信息缺失时应标注“需人工补充”。

5. **不替代本人写作**
   技能包可以帮助整理结构、发现问题、提出格式建议，但论文内容必须由学生本人理解、判断和最终确认。

6. **不用于规避审查**
   可以把表达改得更自然、更具体、更符合本科论文语境，但不能以规避检测、掩盖来源或降低原创性要求为目的。

## 适用对象

适合以下情况：

- 中国矿业大学计算机、软件工程、数据科学、人工智能、信息管理等相关方向本科毕业设计（论文）；
- 已经有真实系统、网站、学习平台、数据库应用、管理系统或算法原型；
- 已经拿到学校模板、任务书、开题报告或往届样文，但不知道如何落到自己的项目；
- 需要检查图表、目录、页码、公式、参考文献和章节结构是否规范；
- 需要根据项目新功能同步更新论文中的需求、实现、截图和测试内容。

不适合以下情况：

- 直接生成一篇可提交论文；
- 缺少真实项目或真实实验支撑时强行拼凑正文；
- 伪造系统工作量、测试结果、参考文献或截图；
- 替代导师意见、学校规范或学生本人判断。

## 仓库内容

技能主文件，定义整体工作边界、输入优先级、工作模式、文档编辑规则、图表公式参考文献要求和学术诚信守则。

### `assets/cumt-thesis-template.doc`

中国矿业大学毕业设计（论文）撰写模板资源。使用时应复制为工作副本，不要直接修改原始模板资产。

### `references/template-spec.md`

整理学校模板中的章节顺序、摘要、目录、图题、表题、公式、参考文献、WPS 使用注意事项等格式规则。

面向真实软件/系统/数据库项目的论文材料整理流程，包含项目事实底稿、证据清单、章节节奏、图表建议、测试章节写法和最终检查清单。

### `agents/openai.yaml`

界面展示相关的名称和默认提示配置，不承载论文事实或学术内容。

## 推荐使用流程

### 1. 收集材料

优先准备：

- 学校论文模板；
- 任务书、开题报告、导师要求；
- 往届优秀样文或学院格式说明；
- 论文题目、专业、学院、导师、学号等封面信息；
- 项目源码、README、接口文档、数据库结构；
- 系统截图、流程图、E-R 图、测试数据、实验图表；
- 已下载并核验的参考文献。

材料不完整时，应先列出“已有材料”和“缺失材料”，不要急于成文。

### 2. 冻结项目事实

写任何正文前，先整理稳定的项目事实底稿，包括：

- 项目目标；
- 使用场景；
- 用户角色；
- 功能模块；
- 前端页面与后端接口；
- 数据库表和关键字段；
- 核心业务流程；
- 算法、规则或模型对比；
- 测试环境、测试用例和测试结果；
- 已知限制和未完成部分。

后续论文内容应围绕这个事实底稿展开，避免前后说法不一致。

### 3. 建立证据清单

建议为每个重要说法建立对应关系：

- 论文中说了什么；
- 对应的代码、截图、表结构、测试记录或文献是什么；
- 出现在论文哪一章、哪张图、哪张表；
- 是否已核验，是否需要学生补充。

这一步可以有效避免“论文写得很完整，但项目里找不到对应证据”的问题。

### 4. 规划目录和图表

计算机类系统论文通常需要：

- 系统总体架构图；
- 核心业务流程图；
- 角色使用流程图；
- 数据库 E-R 图；
- 核心数据库表说明；
- 关键算法或规则流程图；
- 主要页面截图；
- 测试用例表；
- 实验或数据分析图表。

图表不应只用于“凑篇幅”，每张图表都要和正文说明相互对应。

### 5. 按模板检查格式

定稿前重点检查：

- 封面、声明、诚信承诺、授权声明是否保留；
- 中文摘要、英文 Abstract、关键词是否对应；
- 中文目录、英文 Contents 是否更新；
- 页码是否区分前置部分和正文部分；
- 图题是否在图下方，表题是否在表上方；
- 图表编号是否连续；
- 正文是否先引用再放图表；
- 公式是否清晰、编号是否正确；
- 参考文献是否满足 GB/T 7714 格式；
- 致谢、参考文献、附录等是否按要求另起页。

## 最近一次规则增强

本版本根据实际论文整理过程补充了以下规则：

- 增强“真实项目事实优先”的要求；
- 增加项目变更后同步更新论文的流程；
- 增加 WPS 环境下更新目录、域、页码和导出 PDF 的注意事项；
- 增加图表不应整页堆放、截图清晰度、图题中英文命名的检查要求；
- 增加数据库设计中“E-R 图 + 核心表说明表”的推荐做法；
- 增加公式乱码、伪代码、算法表和 Word 公式对象的处理提醒；
- 增加参考文献数量、外文文献数量和 GB/T 7714 信息缺失核查；
- 增加“降低模板化表达”但不规避检测、不改变事实边界的写作要求；
- 强化对原型系统、演示数据、模型对比和 AI 辅助功能的表述边界。

-

使用时仍需注意：技能包只能帮助建立“可写、可证、可核验”的技术分析框架，不能替代学生本人完成论文正文和学术判断。

## 最终提交前建议

提交前建议至少进行一次人工核对：

- 对照学校最新学生手册；
- 对照导师和学院的格式要求；
- 打开 Word/WPS 检查目录、页码、图表和分页；
- 打印或导出 PDF 检查截图清晰度；
- 逐条核对参考文献是否真实、完整、已在正文引用；
- 确认没有占位符、未补充材料或不属于论文正文的工具说明。

如果发现学校要求与本仓库说明冲突，应以学校、学院、导师和当前模板为准。
